# M10 Sentry 연동 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Sentry로 에러 추적(5xx·미처리 예외·outbox poison 격리)과 성능 모니터링(HTTP 트랜잭션, 경로별 샘플링)을 더하되, DSN 없으면 no-op하고 PII는 스크럽한다.

**Architecture:** `src/common/sentry/`에 순수 헬퍼(`sentry-scrub`·`traces-sampler`)와 `initSentry` 래퍼를 두고, main(`main.ts`)·워커 4종 부트스트랩 맨 앞에서 init. HTTP 5xx는 기존 `AllExceptionsFilter`에 `Sentry.captureException`(userId·role·path·method 첨부)을 더하고, 성능은 `tracesSampler` + `SentryModule.forRoot()`로 계측한다. outbox relay는 poison 격리(M9 `{quarantined}`) 시 캡처.

**Tech Stack:** `@sentry/nestjs`(전이 `@sentry/node`), NestJS, Jest. 분산 트레이싱(HTTP→Kafka→워커)은 M10.5로 분리.

> 설계 근거: [M10 설계 스펙](../specs/2026-06-17-m10-sentry-design.md)

## 사전 지식 (실행자가 알아야 할 것)

- **현재 에러 처리:** `src/common/errors/all-exceptions.filter.ts`가 모든 예외를 에러 봉투(M2.5)로 변환하고, `statusCode>=500`에서만 `logger.error`로 스택을 남긴다. 여기에 Sentry 캡처를 더한다(봉투 응답은 불변).
- **인증 사용자:** `req.user`는 `TokenPayload`(`src/auth/domain/token-issuer.ts`) — `sub`(userId)·`email`·`role`. **email은 PII라 Sentry에 안 싣고** `sub`·`role`만.
- **프로세스 구성:** main(HTTP) + 워커 4종(`src/workers/*.main.ts`: persistence·notification·audit·outbox-relay)이 각자 부트스트랩. Sentry는 **각 프로세스에서 따로 init**해야 한다.
- **no-op:** `@sentry/node`는 `dsn`이 없으면 `Sentry.init`을 호출해도(또는 안 해도) 이벤트를 안 보낸다. 우리는 dsn 비면 init 자체를 건너뛴다.
- **config 패턴:** env 키는 `ConfigKey` enum(`src/config/config-keys.ts`) + `.env.example`. 부트스트랩(app 생성 전)에서는 `ConfigService`가 없으니 `process.env[ConfigKey.X]`로 직접 읽는다(enum 참조 → 하드코딩 아님).
- **테스트:** `@sentry/nestjs`를 `jest.mock`으로 막아 외부 전송 없이 호출만 검증.

> **스펙 §5.2 대비 범위 메모:** M10은 **워커 init(프로세스 레벨 자동 캡처: uncaughtException/unhandledRejection) + outbox relay poison 캡처**까지 한다. Kafka 컨슈머 핸들러별 명시 캡처(microservice ExceptionFilter)는 프레임워크 결합이 까다로워 **후속(M10.5)으로 미룬다** — init의 기본 통합으로 프로세스 크래시는 이미 보고된다.

// Sentry beforeSend 훅에서 민감정보를 제거하는 순수 함수.
// sendDefaultPii:false로 헤더 자동 첨부는 막지만, 혹시 실린 민감 헤더를 한 번 더 막는다.
const SENSITIVE_HEADERS = ['authorization', 'cookie'];

interface ScrubbableEvent {
  request?: { headers?: Record<string, unknown> };
  [key: string]: unknown;
}

export function scrubEvent<T extends ScrubbableEvent>(event: T): T {
  const headers = event.request?.headers;
  if (headers) {
    for (const key of Object.keys(headers)) {
      if (SENSITIVE_HEADERS.includes(key.toLowerCase())) {
        headers[key] = '***';
      }
    }
  }
  return event;
}
```

// 트랜잭션 이름(예: "GET /docs")으로 성능 샘플링 비율을 정하는 순수 함수.
// 비즈니스 외 경로(문서 등)는 0으로 두어 노이즈·비용을 줄인다.
// (헬스체크 등이 생기면 EXCLUDED에 추가. CORS preflight는 미사용이라 제외 안 함.)
const EXCLUDED_PATHS = ['/docs']; // '/docs-json'도 includes('/docs')로 함께 잡힌다

export function decideTraceSample(
  name: string | undefined,
  defaultRate: number,
): number {
  if (name && EXCLUDED_PATHS.some((p) => name.includes(p))) {
    return 0;
  }
  return defaultRate;
}
```

```ts
import * as Sentry from '@sentry/nestjs';
import { scrubEvent } from './sentry-scrub';
import { decideTraceSample } from './traces-sampler';

export interface SentryInitOptions {
  dsn: string;
  environment: string;
  tracesSampleRate: number;
}

// Sentry 초기화를 한 곳에 캡슐화. main·워커가 부트스트랩 맨 앞에서 호출한다.
// dsn이 비면 init을 건너뛴다(no-op) → 외부 전송 없음. 활성 여부를 boolean으로 돌려준다.
export function initSentry(opts: SentryInitOptions): boolean {
  if (!opts.dsn) return false;
  Sentry.init({
    dsn: opts.dsn,
    environment: opts.environment,
    sendDefaultPii: false, // 헤더·쿠키·IP 자동 첨부 안 함
    tracesSampler: (ctx) => decideTraceSample(ctx.name, opts.tracesSampleRate),
    beforeSend: (event) => scrubEvent(event),
  });
  return true;
}
```

### 개념
