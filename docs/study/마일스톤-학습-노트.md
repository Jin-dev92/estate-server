## 0. 마일스톤 한눈에 — "무엇을 배우려고 끊었나"

| 단계 | 만든 것 | 핵심 학습 포인트 |
|---|---|---|
| **M0** | 프로젝트 기반 + JWT 인증(Auth 컨텍스트) | Prisma 기초·마이그레이션, 해시·토큰, Passport 전략 |

| **M2** | 게시판 CRUD + Redis 캐시 | read-through 캐시 / 명시적 무효화 / TTL 안전망 |
| **M2.5** | 전역 에러 처리 + 에러 envelope | ExceptionFilter, 안정적 `code` 계약 |
| **M2.6** | Swagger(OpenAPI) 문서화 | 데코레이터 기반 스키마, enum 명명 스키마 |
| **M2.7** | 논리삭제(soft delete) | `deletedAt`, repository 캡슐화, 카스케이드 정책 |
| **M3** | Kafka 도입 + audit-worker | producer/consumer, 멱등 소비, 토픽 사전생성 |
| **M4** | 1:1 채팅(WS+Redis pub/sub+Kafka) | 실시간 전달과 영속화 분리, 쓰기 버퍼, 파티션 키 |
| **M5** | 알림 + 다중 컨슈머 그룹 팬아웃 | 독립 consumer group, 워커 프로세스 분리, 원자적 카운터 |
| **M6** | rate limit(userId+IP 이중) + 보안 점검 | Redis 고정 윈도우 카운터, 전역 가드, 백엔드 이중 제한 |
| **Outbox** | Transactional Outbox(dual-write 유실 제거) | 한 트랜잭션 커밋, 폴링 relay, FOR UPDATE SKIP LOCKED, at-least-once+멱등 |
| **M7** | k6 API 부하테스트 | p95/p99·RPS·thresholds, smoke vs load, closed/open 모델, "측정이 측정을 방해" |
| **M8** | 부하 한계 탐색(stress/spike) | open(arrival-rate) executor, knee·throughput 천장, 병목=DB 풀(P2024), backpressure 위치 |
| **M9** | Outbox DLQ·재시도 백오프 | 지수 백오프(base*2^n, cap), MAX_ATTEMPTS 초과 시 FAILED 격리, at-least-once 유지 |
| **M10** | Sentry(에러 추적·성능) | observability, 에러추적 vs 로깅, 트랜잭션/스팬, tracesSampler, PII 스크러빙, 외부 SaaS 트레이드오프 |
| **M11** | 좋아요 카운터 Redis 전환 + k6 전후 측정 | 파생 캐시 vs 진실 원천, 존재-시-증감(0→1 함정), SET NX 백필, best-effort 쓰기+TTL 치유, 통제 실험(before/after) |
| **CI** | PR 게이트 + 버전 범프 | GitHub Actions, service container, prisma migrate diff(drift), 커밋 타입 기반 버전 산정 |

> **관통 주제 하나만 기억한다면:** "**체감 지연을 낮추기 위해 동기 경로를 짧게 하고, 무거운 일(영속화·알림·감사)은 이벤트로 비동기화한다.** 단, 비동기는 정합성 구멍을 만들므로 **멱등**과 **결과적 정합성**으로 메운다."

## 1. Prisma & 관계형 모델링 (M0·M1·M2.7)

### 개념
- `건물(Building) → 호실(Unit) → 입주(Lease)` 3계층을 1:N / N:1 관계로 모델링. `User`는 OWNER/TENANT 역할을 가지며 여러 관계의 끝점이 된다.
- 마이그레이션 워크플로우: `prisma migrate dev`(개발, 스키마→SQL 생성+적용) vs `prisma migrate deploy`(운영, 생성된 마이그레이션만 적용).
- `String` → PostgreSQL `text`, PK는 `@default(cuid())`.

### 왜 중요
- 2계층(건물→입주)으로 줄이면 "특정 호실 입주자에게만"이라는 도메인 의미가 사라진다. **모델의 무게 = 학습 표면적**이라고 보고 일부러 3계층을 유지했다.
- 같은 모델을 두 FK가 가리키면 **named relation**이 필요하다(예: ChatRoom이 owner/tenant 둘 다 User를 가리킬 때).

### 우리 코드
- [`prisma/schema.prisma`](../../prisma/schema.prisma) — 모델·인덱스·유니크.
- [`src/property/`](../../src/property) — Property 컨텍스트(건물·호실·입주).

### 스스로 점검
- [ ] `migrate dev`와 `migrate deploy`의 차이를, "왜 운영에선 deploy인가"까지 설명할 수 있는가?
- [ ] `@@unique([a, b])`와 `@@index([a, b])`는 각각 언제 쓰며, 조회 성능에 어떻게 작동하는가?
- [ ] `cuid`와 `uuid`의 차이, FK를 `text`로 둔 트레이드오프는?

### 더 팔 키워드
`복합 인덱스 카디널리티`, `N+1 문제와 include/select`, `트랜잭션 격리수준`, `Prisma Client Extension(자동 필터)`

## 2. Redis — 한 저장소, 여러 용례 (M1·M2·M4·M5·M6)

이 프로젝트는 Redis를 **6가지 용례**로 쓴다(아래 표). "Redis = 캐시"라는 1차원 이해를 깨는 게 목표.

| 용례 | 어디서 | 핵심 |
|---|---|---|
| **TTL 만료** | 초대코드(M1) | `SET key val EX <초>` — 코드가 자동 만료 |
| **read-through 캐시 + 명시적 무효화** | 게시판(M2) | 읽기 시 캐시 채우고, 쓰기 시 키 삭제 + 짧은 TTL 안전망 |
| **pub/sub 중계** | 채팅·알림(M4·M5) | 단일 채널 발행 → 모든 인스턴스가 구독 → 로컬 소켓에 emit |
| **capped list 캐시** | 채팅 최근 메시지(M4) | `LPUSH` + `LTRIM 0 N-1`로 최근 N개만 유지 |
| **원자적 카운터** | 미읽음 수(M5) | `INCR`/`DEL` — 동시성에도 카운트 유실 없음 |
| **rate limit(고정 윈도우)** | RateLimitGuard(M6) | `ratelimit:{scope}:{id}:{창}` 키에 Lua로 `INCR`+최초 1회 `EXPIRE`(원자), `count>한도`면 429 |

### 왜 중요
- **pub/sub 전용 연결:** 구독 모드 연결은 일반 명령을 못 쓴다 → `redis.duplicate()`로 구독 전용 커넥션을 따로 만든다. ([`redis-message-relay.ts`](../../src/chat/infrastructure/redis-message-relay.ts), [`redis-notification-relay.ts`](../../src/notification/infrastructure/redis-notification-relay.ts))
- **캐시 무효화 타이밍**이 캐시의 본질적 난제다. 우리는 "쓰기 시 명시적 삭제 + 짧은 TTL 안전망"으로 stale 위험을 이중으로 막았다.
- **카운터는 비정규화 캐시**다. `미읽음 수 = COUNT(*) WHERE readAt IS NULL`의 캐시이므로, Redis가 eviction되면 유실된다 → no-eviction 인스턴스 가정을 코드 주석에 남겼다. ([`redis-notification-counter.ts`](../../src/notification/infrastructure/redis-notification-counter.ts))
- **rate limit은 Redis의 대표 용례**다(M6). `ratelimit:{scope}:{id}:{창번호}` 키를 윈도우마다 `INCR`하고 한도와 비교한다. **`INCR`과 `EXPIRE`를 한 Lua로 묶는 이유**: 분리하면 "INCR 후 EXPIRE 직전 크래시" 시 TTL 없는 영구 키가 남아 카운터가 영원히 초과 상태가 된다. ([`rate-limit.store.ts`](../../src/common/rate-limit/rate-limit.store.ts), [`rate-limit.guard.ts`](../../src/common/rate-limit/rate-limit.guard.ts)). **rate limit ≠ debounce/throttle**: 후자는 클라이언트에서 호출을 조용히 줄이는 UX 기법, 전자는 서버에서 초과를 429로 **거부**하는 보호 정책. 또 **앱 레벨 rate limit ≠ DDoS 방어**(분산 공격은 CDN/WAF/네트워크 계층의 일).

### 스스로 점검
- [ ] pub/sub로 "멀티 인스턴스에서도 실시간 전달"이 되는 원리를 그림으로 그릴 수 있는가? (워커는 소켓이 없는데 어떻게 푸시가 도달하는가?)
- [ ] 캐시 무효화에서 "쓰기 후 삭제" vs "쓰기 후 갱신(write-through)"의 트레이드오프는?
- [ ] `INCR`이 왜 race-safe인가? (단일 스레드 이벤트 루프 + 원자적 명령)
- [ ] capped list를 `LPUSH`+`LTRIM`로 만들 때 순서(최신이 앞)와 조회 방향은?
- [ ] 고정 윈도우의 '경계 2배 버스트'가 왜 생기는지, 슬라이딩 윈도우가 어떻게 완화하는지 설명할 수 있는가?
- [ ] `INCR`과 `EXPIRE`를 왜 한 Lua로 묶나? (분리 시 TTL 없는 영구 키 race)
- [ ] 프론트엔드 rate limit만으로 부족한 이유(우회 가능)와 백엔드 userId+IP 이중 제한의 의미는?
- [ ] rate limit / debounce / throttle / DDoS 방어의 차이를 한 문장씩으로 구분할 수 있는가?

### 더 팔 키워드
`cache stampede / thundering herd`, `Lua 스크립트로 원자적 멀티 연산`(이미 [`redis.service.ts`](../../src/redis/redis.service.ts)에 `runScript`), `Redis Streams vs pub/sub`, `maxmemory-policy`

## 3. Kafka — 이벤트 드리븐의 핵심 (M3·M4·M5)

### 3.1 producer / consumer 첫걸음 (M3)
- 도메인 변경이 일어나면 **도메인 이벤트**(`PostCreated`, `MessageSent` 등)를 Kafka에 발행한다.
- 발행은 application 레이어가 `EVENT_PUBLISHER` 포트를 통해 한다 → **도메인은 Kafka를 모른다**(의존성 역전).
- 소비자(worker)는 `@EventPattern(topic)`으로 토픽을 구독해 부수효과(적재)를 수행한다.
- 코드: [`src/events/`](../../src/events) (이벤트 envelope·발행자·토픽), [`audit-worker.controller.ts`](../../src/audit/interface/audit-worker.controller.ts)

### 3.2 멱등 소비 (at-least-once 대응)
- Kafka는 **at-least-once** — 같은 메시지가 두 번 올 수 있다. 그래서 소비자는 **멱등**해야 한다.
- 기법: **앱이 생성한 ID를 PK/유니크 키로** 두고, 중복 INSERT 시 `P2002`(unique 위반)를 잡아 무시.
  - audit: `eventId @unique` ([`prisma-audit-log.repository.ts`](../../src/audit/infrastructure/prisma-audit-log.repository.ts))
  - 메시지: `Message.id = messageId` ([`prisma-message.repository.ts`](../../src/chat/infrastructure/prisma-message.repository.ts))
  - 알림: `@@unique([eventId, recipientId])` ([`prisma-notification.repository.ts`](../../src/notification/infrastructure/prisma-notification.repository.ts))

### 3.3 토픽 사전생성 (production 모사)
- 운영에선 `auto.create.topics.enable=false`(오타 토픽 난립·파티션 수 미제어 방지). 우리도 끄고, 부팅 시 [`KafkaTopicInitializer`](../../src/events/kafka-topic-initializer.ts)가 `listTopics` 차집합만 생성(멱등).
- **콜드스타트 race:** 컨슈머가 구독을 시작할 때 토픽이 없으면 "This server does not host this topic-partition" → `startAllMicroservices()` 전에 `ensureTopics()`를 명시적으로 await해 해결.

### 3.4 다중 컨슈머 그룹 팬아웃 (M5) — 이번 마일스톤의 핵심
- **consumer group의 의미:** 같은 group 안에선 파티션을 나눠 갖고(부하 분산), **다른 group은 같은 메시지를 각자 한 번씩** 받는다(팬아웃).
- 우리는 이벤트 1건을 **persistence·notification·audit 세 group이 독립 소비**하게 했다.
- **NestJS 한계와 해법:** hybrid 앱에서 `@EventPattern` 핸들러는 연결된 모든 마이크로서비스에 전역 등록된다 → 그룹별 분리가 안 된다. 그래서 **워커별 엔트리포인트**로 프로세스를 쪼갰다([`src/workers/`](../../src/workers)): `NestFactory.create` 후 `listen()` 없이 `connectMicroservice`(groupId 1개)만.

```
도메인 이벤트 1건 (board-events / chat-events)
        ├─ persistence-worker  group → Message 적재
        ├─ notification-worker group → Notification 적재 + WS 푸시
        └─ audit-worker        group → AuditLog 적재
   (세 group이 같은 메시지를 각각 한 번씩 = 진짜 팬아웃)
```

### 3.5 쓰기 버퍼 & 파티션 키 (M4)
- 채팅 메시지를 동기 INSERT 대신 `chat-events`로 흘려 **쓰기 스파이크를 Kafka가 흡수**(쓰기 버퍼).
- **순서 보장:** 파티션 키 = `roomId` → 같은 방 메시지는 같은 파티션으로 가 순서가 보장된다. (파티션이 곧 순서 단위)

### 스스로 점검
- [ ] "consumer group이 같다 vs 다르다"가 메시지 분배에 어떤 차이를 만드는가? 파티션 수와 컨슈머 수의 관계는?
- [ ] at-least-once / at-most-once / exactly-once의 차이와, 우리가 at-least-once + 멱등을 택한 이유는?
- [ ] 파티션 키를 잘못 잡으면(예: 전부 같은 키) 무슨 문제가 생기나?
- [ ] 토픽을 미리 만들지 않으면 콜드스타트에 무슨 에러가 났고, 왜 라이프사이클 훅이 아니라 명시적 await로 풀었나?

## 4. 실시간 전달 ↔ 영속화 분리 (M4·M5) — 아키텍처 사고의 핵심

### 개념 (한 문장)
> 사용자에게는 **Redis pub/sub로 즉시** 전달하고, **DB 적재는 Kafka를 거쳐 비동기**로 한다. 둘을 분리해 체감 지연을 낮추고 쓰기 스파이크를 흡수한다.

### 데이터 흐름 (채팅 메시지 1건)
1. WS Gateway가 메시지 수신(핸드셰이크 JWT 인증)
2. Redis 단일 채널에 발행 → 모든 인스턴스가 구독 → 상대방 소켓에 **즉시 emit** (체감 지연 최소)
3. Redis capped list에 push(최근 메시지 캐시)
4. Kafka `chat-events`에 발행 → persistence-worker가 **비동기 멱등 INSERT**

### 트레이드오프(꼭 이해할 것)
- "**전달은 됐는데 DB엔 아직 없는**" 짧은 윈도우가 생긴다(결과적 정합성). 학습 수준에선 허용, 엄밀하게는 **Transactional Outbox**(Outbox 마일스톤, M6 이후)로 푼다.
- 알림 푸시(M5)는 워커(별도 프로세스)가 소켓을 못 가지므로, **Redis 채널 → main의 Gateway**로 브리지해 emit한다. 푸시는 **best-effort**(적재·카운터가 진실 원천이므로 푸시 실패가 적재를 막지 않게 try/catch). ([`handle-event.use-case.ts`](../../src/notification/application/handle-event.use-case.ts))

### 우리 코드
- [`chat.gateway.ts`](../../src/chat/interface/chat.gateway.ts), [`notification.gateway.ts`](../../src/notification/interface/notification.gateway.ts) — WS 핸드셰이크 인증·룸·구독.

### 스스로 점검
- [ ] WS 핸드셰이크에서 JWT를 어떻게 검증하고, 실패 시 어떻게 처리하나?
- [ ] "전달됐는데 DB엔 아직" 윈도우가 실제로 사용자에게 어떤 증상으로 나타날 수 있나? Outbox는 이를 어떻게 없애나?
- [ ] 왜 알림 푸시는 best-effort이고 적재는 그렇지 않은가? (진실 원천 개념)

### 더 팔 키워드

## 7. 견고함을 위한 횡단 관심사 (M2.5·M2.6·M2.7)

### 7.1 전역 에러 처리 + 에러 envelope (M2.5)

> **envelope(에러 응답 포장 형식)란?** 안에 든 에러가 무엇이든 **겉 JSON 구조는 항상 똑같이** 감싸는 표준 "겉포장" 양식을 말한다(내용물이 달라도 똑같은 규격 상자에 담아 보내는 것처럼). 예:
> ```json
> {
>   "statusCode": 404,
>   "code": "BOARD_POST_NOT_FOUND",
>   "message": "게시글을 찾을 수 없습니다.",
>   "path": "/posts/abc123",
>   "timestamp": "2026-06-12T08:00:00.000Z"
> }
> ```
> 게시글 없음·권한 없음·검증 실패 등 **종류와 무관하게 이 5필드 구조**로 내려간다.

- 모든 4xx/5xx를 `AllExceptionsFilter`가 위 **단일 envelope**로 통일한다.
- **FE는 메시지 문구가 아니라 안정적인 `code`로 분기**한다 — `message`("게시글을 찾을 수 없습니다")는 바뀔 수 있지만 `code`(`BOARD_POST_NOT_FOUND`)는 변하지 않는 **계약**이기 때문.
- 커스텀 예외 `AppException(BoardError.POST_NOT_FOUND)` → 필터가 envelope로 변환.
- 코드: [`src/common/errors/`](../../src/common/errors) · 전체 code 표는 [README §7 "에러 응답 형식"](../../README.md).

### 7.2 Swagger / OpenAPI (M2.6)
- 모든 엔드포인트에 `@ApiTags`·`@ApiOperation`·`@ApiResponse`, 보호 라우트엔 `@ApiBearerAuth`.
- **enum은 `@ApiProperty({ enum, enumName })`** 로 명명 스키마 노출(허용값을 문서가 강제).
- 4xx는 `@ApiResponse({ type: ErrorResponseDto })`로 envelope 계약 표기.

### 7.3 논리삭제 soft delete (M2.7)
- 5개 엔티티(`User·Building·Unit·Post·Comment`)에 nullable `deletedAt`. 조회는 repository에서 `deletedAt: null` 필터를 **캡슐화**(도메인·유스케이스는 soft delete를 모름).
- `Lease`는 이미 `status(ACTIVE/ENDED)`로 "종료"를 표현 → 의미 중복 회피로 제외.
- **카스케이드:** 물리삭제가 사라지면 DB `onDelete: Cascade`가 무의미 → Post soft delete 시 자식 Comment를 **같은 트랜잭션에서 앱 레벨로 함께** soft delete.

## 8. 멱등성 & 결과적 정합성 — 비동기 시스템의 두 기둥 (M3·M4·M5·Outbox)

이 두 개념이 M3~M5와 Outbox를 관통한다. **따로 묶어 복습할 가치**가 있다.

### 멱등성(Idempotency)
- "같은 작업을 여러 번 해도 결과가 한 번 한 것과 같다." Kafka at-least-once 때문에 **필수**.
- 패턴: **앱 생성 ID를 유니크 키로** → 중복 시 `P2002` 잡고 무시. 카운터처럼 부수효과가 있으면 **"신규 삽입에 성공했을 때만"** 증가시켜 중복 증가를 막는다([`handle-event.use-case.ts`](../../src/notification/application/handle-event.use-case.ts)의 `if (!created) continue;`).

### 결과적 정합성(Eventual Consistency)
- 즉시 일관이 아니라 "잠시 후 일관." 우리 시스템의 윈도우들:
  - 전달됐는데 DB엔 아직(채팅) → 본질은 dual-write, **Outbox로 해소**.
  - 미읽음 카운터와 행 수의 드리프트(읽음 처리와 워커 INCR 경합) → 코드 주석에 명시, DB COUNT 폴백이 정공법.

### Transactional Outbox — dual-write를 없애는 법
- **문제(dual-write):** use case가 `repo.create()`(트랜잭션 1)를 커밋한 뒤 별도로 `events.publish()`(Kafka)를 호출하면, 그 사이 크래시 시 "DB는 썼는데 이벤트 유실".
- **해법:** 도메인 변경 + `OutboxEvent` 행 INSERT를 **한 트랜잭션**으로 커밋(`txRunner.run(tx => { repo.create(.., tx); outbox.add(event, tx) })`). DB가 커밋되면 outbox 행도 반드시 있다 → 유실 불가. 별도 **relay 워커**가 PENDING을 폴링해 Kafka 발행 후 PUBLISHED 마킹. ([`relay-outbox.use-case.ts`](../../src/outbox/application/relay-outbox.use-case.ts), [`prisma-outbox-store.ts`](../../src/outbox/infrastructure/prisma-outbox-store.ts))
- **멀티 relay 안전:** `SELECT … FOR UPDATE SKIP LOCKED`를 트랜잭션 안에서 써, 여러 relay가 동시에 돌아도 같은 행을 잡지 않는다(잠금은 트랜잭션 동안 유효).
- **핵심 한계:** Outbox는 **"유실 없음"은 주지만 "중복 없음"은 못 준다**. relay 재시도·멀티 relay로 같은 이벤트가 두 번 발행될 수 있어 결국 **at-least-once** → 소비자 멱등(`eventId @unique`)이 필수 짝꿍이다.

### 후속: DLQ(Dead Letter Queue) — "영원히 안 되는 이벤트" 격리 (미구현)
- **지금 동작:** relay는 발행 실패 시 `attempts++` 하고 status는 PENDING 유지 → 다음 폴링에 **또 시도**(영원히 재시도). 대부분의 실패는 일시적(Kafka 잠깐 다운)이라 이게 정답이다.
- **문제(poison message):** payload 손상·없는 토픽처럼 **절대 성공 못 할** 이벤트가 끼면, 매 폴링마다 실패하며 로그를 더럽히고 배치 앞을 막을 수 있다(head-of-line blocking).
- **DLQ = "N번 실패하면 따로 빼두는 곳"**(배달 못 한 편지를 반송함에 모으듯). 우리 규모의 가장 단순한 형태는 **상태 하나 추가**: `attempts ≥ 임계(예: 5)`면 `PENDING → FAILED`로 전환. relay는 `WHERE status='PENDING'`만 집으니 FAILED는 자동 재시도에서 빠진다 → 정상 흐름을 안 막음. 운영자는 FAILED만 조회해 원인 보고, 고친 뒤 PENDING으로 되돌려 재처리.
- **왜 미뤘나(축: 견고함 ↔ 단순함/YAGNI):** Outbox의 핵심 목표인 "유실 제거"는 이미 달성. DLQ는 그 위에 얹는 **운영 견고함**이고, 무한 재시도는 일시 장애엔 옳은 동작이라 당장 위험하지 않아 분리했다.

### M9 — DLQ·재시도 백오프 (Outbox 견고화)
- **문제:** 기존 relay는 실패 시 status를 PENDING으로 둬 매 틱(1s) 무한 재시도 → poison message가 영영 안 빠지고 Kafka 호출·로그를 낭비.
- **해법:** ① 실패 시 `nextAttemptAt = now + computeBackoff(attempts)`로 **지수 백오프**(`base*2^n`, cap 60s) → `fetchPending`이 `nextAttemptAt > now`인 행을 제외해 즉시 재시도가 사라짐. ② `attempts`가 `MAX_ATTEMPTS`(기본 5) 도달 시 `PENDING → FAILED`로 **격리**(DLQ) → 더는 폴링 안 됨. `lastError`/`failedAt`가 사후 조사 근거.
- **트레이드오프:** 무한 재시도(언젠가 성공) ↔ 격리(정상 흐름 보호+사람 개입 전제). 즉시 재시도(빠른 복구) ↔ 백오프(부하 절감, 복구 지연). **at-least-once는 그대로** — 백오프·격리는 "유실 없음"을 안 바꾸고, 중복 발행도 여전히 가능(소비자 멱등이 흡수). 격리는 "무한 실패 행"만 빼낼 뿐.
- **범위 밖(후속):** replay(FAILED→PENDING 되살리기), 메트릭/알림(M10 Sentry).

#### 용어 풀이 (처음 보면 헷갈리는 것들)
- **DLQ (Dead Letter Queue, 데드 레터 큐):** "죽은 편지함". 우체국에서 수취인 불명으로 못 배달한 편지(dead letter)를 따로 모아두던 데서 온 말. 메시지 큐에서 **반복 실패한 메시지를 정상 흐름에서 빼내 격리하는 곳**을 뜻한다. 격리된 메시지는 사람이 나중에 들여다보고(원인 조사) 고치거나 재처리한다. *우리 구현:* 별도 큐를 만들지 않고 Outbox 행의 `status`를 `FAILED`로 바꾸는 것 자체가 DLQ 역할(테이블 컬럼으로 같은 목적 달성).
- **poison message (독약 메시지):** 스키마 불일치·영구 오류 등으로 **몇 번을 재시도해도 영원히 실패하는 메시지**. 정상 큐에 두면 매번 재시도하며 자원·로그를 낭비하고 뒤 메시지를 막는다 → DLQ로 격리해야 하는 대상이 바로 이것.
- **지수 백오프 (exponential backoff):** 재시도 간격을 실패할수록 **2배씩 늘리는** 전략(1s→2s→4s→8s…). 일시 장애엔 빨리 복구하되, 지속 장애엔 호출 빈도를 지수적으로 줄여 부하·로그 폭주를 막는다. 무한히 커지지 않게 **cap(상한)** 을 둔다. "linear(고정 간격)"보다 장애 대응에 유리.
- **재시도 vs 격리의 관계:** 백오프로 N번 재시도 → 그래도 안 되면(= poison으로 판단) DLQ 격리. 즉 백오프는 "기다렸다 다시", 격리는 "이제 그만, 사람 불러".

## 8.5 부하테스트 (M7·M8) — 측정의 기술

성격이 다른 대표 엔드포인트 4개를 [k6](../../load/)로 재 baseline을 잡았다.

### 핵심 개념
- **왜 평균이 아니라 p95/p99인가:** 평균은 꼬리 지연을 숨긴다. "사용자의 5%가 느리다"가 SLO에서 중요 → 백분위로 본다. (우리 실측: read p95 6.9ms, create 19.6ms, **login(bcrypt) 114ms** — 인증이 CPU 바운드라는 걸 숫자로 확인.)
- **check vs threshold:** check는 개별 응답 검증(실패해도 계속), threshold는 합격/불합격 게이트(미달 시 k6 exit≠0 → CI red). 역할 분리.
- **think time(sleep):** 넣으면 *현실적 동시 사용자*, 빼면 *최대 RPS 한계*. 축: 현실성 ↔ 최대 처리량. baseline엔 think time 있는 쪽이 정직.
- **closed(VU) vs open(arrival-rate) 모델:** VU는 응답을 기다렸다 다음 요청 → 시스템이 느려지면 부하가 자동으로 줄어 backpressure가 숨는다(baseline에 충분). arrival-rate는 시스템 속도와 무관히 밀어붙여 한계·적체를 노출(stress/spike에 적합 — M8에서 사용).
- **smoke / load / stress / spike:** 정상성 / baseline / 한계점 탐색 / 급증 충격. smoke·load는 M7, stress·spike는 M8에서 다뤘다(아래).

### stress / spike — "얼마나 버티나 / 충격을 견디나" (M8 구현)
부하의 *목적*이 다른 4종류. smoke·load는 "정상 범위에서 어떤가"를, stress·spike는 "한계와 회복"을 본다. M8에서 뒤 둘을 open(arrival-rate) executor로 구현했다(실측은 아래 "M8 실측 발견").

| 종류 | 부하 모양 | 무엇을 알려고 |
|---|---|---|
| smoke | 아주 약하게(VU 1) | 스크립트·기능이 도나? |
| **load**(우리가 함) | 예상 수준 유지 | 정상 트래픽 p95(= baseline) |
| **stress** | 계속 늘림(20→50→100…) | **어디서 무너지나**(한계점·병목) |
| **spike** | 갑자기 확 쏨(0→500 순간) | **급증 충격을 견디고 회복하나** |

- **stress = 한계점(breaking point) 찾기.** 부하를 계속 올리면 어느 지점에서 p95 폭증·에러 발생 → 그게 한계 용량이고, **먼저 터지는 자원이 병목**이다(우리 후보: DB 커넥션 풀 고갈, bcrypt CPU 포화, Redis/Kafka 연결 한계). 용량 계획의 근거.
- **spike = 급증·회복 테스트.** 평소 20명인데 순간 500명 → ① 죽지 않고 막아내나(rate limit·큐가 활약), ② 스파이크 후 **정상으로 회복**하나(밀린 게 계속 밀리지 않나).
- **환경 한계 ↔ 통제 실험(M8이 택한 길):** stress/spike는 시스템을 진짜 한계까지 밀어야 의미가 있는데, k6·앱·DB가 **한 노트북**에서 도니 "앱이 아니라 내 머신이 먼저 한계"가 된다 → 숫자가 estate-server의 한계가 아니라 측정 환경의 한계. **별도 부하 머신** 대신, M8은 **DB 커넥션 풀을 일부러 1로 좁혀**(`connection_limit=1`) *머신보다 앱이 먼저, 예측한 지점에서* 터지게 만드는 **통제 실험**으로 우회했다. 숫자의 절대성은 포기하되 "병목이 어떻게 드러나는가"는 그대로 학습. (closed VU로는 backpressure가 숨으니, 한계 탐색은 **open/arrival-rate executor**로 갔다.)

### M8 실측 발견 (stress / spike)
- **stress — 병목은 latency로 먼저 드러난다.** 풀을 좁히며 부하를 올리자 p95가 **13.6ms → 210ms → 1734ms**로 폭증. 바꾼 변수는 `connection_limit` 하나뿐이라 병목이 **DB 커넥션 풀**임을 상관으로 확정. throughput은 풀 용량(풀=1 → **~95 RPS**)에서 평평해진다 = "RPS를 더 줘도 안 오르는 천장"이 곧 용량.
- **병목을 이름으로:** 충분히 오래 밀자 앱 로그에 `Timed out fetching a new connection from the connection pool (connection limit: 1)`(Prisma **P2024**)가 35건 → 500. 단 이 타임아웃은 **큐 대기가 `pool_timeout`을 넘겨야** 떠서, 짧게 밀면 latency만 오르고 타임아웃은 안 난다(경계에서 자기안정화).
- **open 모델의 backpressure 위치:** maxVUs가 모자라면 초과분이 `dropped_iterations`로 버려진다 → 부하가 앱이 아니라 **k6 쪽에 쌓인다**. 한계 탐색 땐 maxVUs를 넉넉히 줘야 앱 큐가 깊어져 진짜 한계가 보인다.
- **spike — 방어가 작동하면 급증이 싸진다.** 5→300 RPS 급증에서 초과분 **4032건 429 차단** / 한도 내 **743건 2xx** / **5xx=0**(앱 생존), p95 **10ms**(429 거부는 Redis O(1)이라 거의 공짜 — 진짜 일을 안 하므로 가볍다). 고정 윈도우라 **회복은 다음 윈도우(≤10s) 경계**에서: 스파이크 후 정상 글작성 201 확인. k6 실패율 84%는 *의도된 방어*이지 실패가 아니다.

### 실측에서 배운 것 (함정 2개)
- **"측정이 측정을 방해":** login 부하가 ~99% 429였다. login 라우트의 `@RateLimit({ipMax:10})`이 **데코레이터 하드코딩**이라 `RATE_LIMIT_*` env 상향으로 안 풀린다 → 부하가 곧장 rate limit에 막힘. 순수 bcrypt는 **smoke(윈도우당 ≤10회)**로 따로 쟀다. *우리가 만든 보안이 의도대로 동작한다는 증거이기도 하다.*
- **캐시 과대평가:** read 시나리오가 모든 VU 같은 building을 읽어 Redis hit이 ~100% → 수치가 캐시 최상 시나리오다. 현실적 hit/miss를 보려면 키를 다양화해야 한다.
- **로컬 부하의 한계:** k6+앱+DB가 한 머신에서 자원을 다툼 → 측정값은 **절대치가 아니라 상대 비교·회귀 감지용**.

## 8.6 관측성 — Sentry (M10)

### 개념
- **에러 추적 ≠ 로깅:** 로그는 텍스트 스트림, Sentry는 **그룹핑·빈도·영향 사용자·알림**을 주는 구조화 추적. M2.5 에러 봉투(사용자용)와 Sentry(개발자용)는 역할이 다르다.
- **무엇을 보낼지 통제:** 5xx만(4xx·도메인 예상 예외 제외)·경로별 샘플링(/docs 0)·일시 실패 제외(outbox는 poison 격리만) → 노이즈와 비용(이벤트·트랜잭션 쿼터)을 줄인다. "전부 보내기"는 대시보드를 못 쓰게 만든다.
- **4xx는 비율로 본다:** 개별 4xx는 사용자발(만료 토큰·오타)이 섞여 노이즈지만, **FE/계약 버그**(배포 직후 422·400 폭증)는 진짜 신호다. 그래서 `422·400-validation`만 `SENTRY_4XX_SAMPLE_RATE`(기본 0)로 **저샘플 캡처**(level=warning)하는 옵션을 뒀다 — 평소 끄고 배포 직후·디버깅 때만 켠다. 메서드(GET/POST)가 아니라 **코드 성격**으로 거른다. (진짜 FE 버그 추적의 본류는 FE-side Sentry + 4xx 발생률 알림.)
- **컨텍스트:** userId(sub)·role을 첨부해 "누구에게 난 에러"인지 추적. email은 PII라 제외.
- **성능 모니터링:** HTTP 요청을 트랜잭션으로 계측(M7 부하측정의 운영판). `tracesSampler`로 경로별 비율(메서드가 아니라 경로 성격으로 거른다 — GET도 수집).
- **no-op·PII:** DSN 없으면 비활성, `sendDefaultPii:false` + beforeSend로 Authorization/Cookie 스크럽.
- **구현 메모:** `@sentry/nestjs` v10은 `SentryModule.forRoot()`가 없고, `initSentry()`의 nestIntegration/httpIntegration이 자동 계측. 워커는 init만으로 uncaughtException/unhandledRejection이 프로세스 레벨에서 자동 캡처된다.

### 트레이드오프
- 관측성 ↔ 외부 SaaS 의존(DSN 관리·PII 스크러빙 책임). DSN은 env로만.

### 후속(M10.5) ✅ → §8.9
- 분산 트레이싱(HTTP→Kafka→워커): 트레이스 컨텍스트를 Kafka 헤더·Outbox `traceContext` 컬럼에 실어 프로세스 경계를 넘어 잇는다. 상세는 §8.9.

- [ ] 왜 4xx·도메인 예외는 Sentry로 안 보내나? 다 보내면 무엇이 나빠지나?
- [ ] tracesSampler로 경로를 0으로 두는 이유와, 메서드(GET/POST)로 끄지 않는 이유는?

## 8.7 좋아요 카운터 Redis 전환 — 파생 캐시와 통제 실험 (M11)

### 개념
- **진실 원천(source of truth) vs 파생 캐시:** `PostLike` 테이블(좋아요 행)이 진실 원천이고, Redis의 좋아요 수 카운터는 그로부터 **파생된 캐시**다. 파생 캐시는 (1) 사라져도 원본에서 재구성 가능해야 하고 (2) 원본과 잠시 어긋나는(drift) 것을 설계상 감수해야 한다 — 이 두 조건이 없으면 캐시가 아니라 "두 번째 진실"이 되어 정합성 책임이 배가된다.
- **왜 카운터가 필요했나:** 좋아요 수 조회를 매번 `SELECT COUNT(*) ... GROUP BY postId`로 계산하면, 좋아요 총량이 늘수록 집계 비용이 커진다(이번 실측에서 글당 2000개 좋아요 시 p95가 0개 대비 약 6.9배). Redis `INCR`/`DECR`는 좋아요 총량과 무관한 O(1) 연산이라 이 비용을 없앤다.
- **"존재-시-증감"과 0→1 함정:** 카운터 증감(Lua 스크립트)은 **키가 이미 존재할 때만** 원자적으로 INCR/DECR한다. 만약 키가 없는 상태(아직 캐시가 채워지지 않아 "0"이어야 하는 상태)에서 무조건 INCR부터 했다면, 실제로는 좋아요가 이미 여러 개 있는데도 카운터가 1부터 시작해버리는 **"0→1 함정"**에 빠진다 — 캐시 미스와 최초 좋아요를 구분하지 못하기 때문이다. 그래서 증감은 "존재할 때만" 하고, 없을 땐 아래 백필 경로가 실제 값을 채우도록 역할을 분리했다.
- **SET NX 백필과 경합:** 카운터 미스(키 없음) 시 `readMany`/`readOne`이 `COUNT`로 실제 값을 구해 `SET NX`(키가 없을 때만 설정)로 채운다. 동시에 여러 요청이 같은 미스를 겪어도(레이스 컨디션), `NX` 덕분에 **가장 먼저 도착한 SET만 반영**되고 나머지는 조용히 무시된다 — 여러 스레드가 서로 다른 시점의 COUNT 값으로 덮어써 값이 흔들리는 것을 막는 장치다.
- **best-effort 쓰기 + TTL 치유:** 카운터 증감은 좋아요/취소 트랜잭션이 **커밋된 이후에만** 수행하고, 이 증감 자체가 실패해도 원래 요청(좋아요/취소)은 이미 성공한 것으로 처리한다(best-effort). 그 대신 카운터 키에 **TTL 3600초**를 걸어, 드리프트가 쌓여도 최대 1시간 뒤엔 키가 만료되고 다음 조회에서 COUNT로 재구축되며 자연 정정된다. "항상 정확함" 대신 "일정 시간 내 정확함으로 수렴"을 택한 결과적 정합성(eventual consistency, §8 참고) 설계다.
- **통제 실험(before/after):** 같은 코드베이스에서 구현 전(라이브 COUNT)과 구현 후(Redis 카운터) 두 커밋을 같은 k6 시나리오·같은 시드 볼륨(글 50개 × 좋아요 0/200/2000)으로 측정해 비교했다. 변수를 "카운터 유무" 하나로만 좁혀야 성능 차이의 원인을 그 변수로 귀속시킬 수 있다(M8 stress에서 DB 풀만 좁힌 것과 같은 원리).

### 실측 결과와 해석
| 글당 좋아요 | before(COUNT) p95 | after(Redis 카운터) p95 |
|---|---|---|
| 0 | 10.61ms | 12.20ms |
| 200 | 19.53ms | 12.39ms |
| 2000 | 73.46ms | 13.27ms |

- **볼륨에 대한 민감도가 핵심 차이다.** before는 0→2000 구간에서 p95가 약 6.9배(10.61→73.46ms) 증가하지만, after는 같은 구간에서 약 8.8%(12.20→13.27ms)만 증가해 사실상 평평하다. 카운터 전환의 본질은 "항상 더 빠르다"가 아니라 **"응답 시간이 데이터 총량과 무관해진다"**는 것.
- **저볼륨에서는 역전될 수 있다.** 볼륨 0에서는 오히려 before(10.61ms)가 after(12.20ms)보다 낮다 — Redis 조회 자체의 네트워크 왕복 비용이 고정비로 붙기 때문이다. 좋아요가 거의 없는 글에서는 COUNT가 근소하게 더 유리할 수 있다는 뜻으로, "캐시가 항상 이긴다"는 통념을 실측으로 반증한 사례다.
- **교차점은 200~2000 사이.** 200개에서 이미 after(12.39ms)가 before(19.53ms)를 앞서고, 2000개에서는 격차가 약 5.5배(73.46 vs 13.27ms)까지 벌어진다. 이 프로젝트 규모(로컬 단일 머신, 글 50개)에서도 그 경향이 뚜렷했다.

## 8.8 CI — PR 게이트 + 버전 범프

### 개념
- **PR 게이트:** 머지 전 `pull_request`에서 자동 검증. ① `nest build`(타입체크 — 컴파일 에러를 사람보다 먼저 잡음) ② Prisma **drift 체크**.

## 8.9 분산 트레이싱 — 프로세스 경계를 넘는 trace 전파 (M10.5)

### 개념
- **trace와 span:** **trace**는 하나의 논리적 작업(예: "댓글 작성" 요청 하나)이 여러 서비스를 거치며 남기는 전체 기록이고, **span**은 그 안의 한 작업 단위(HTTP 처리, Kafka 발행, 워커 소비 등)다. 여러 span이 부모-자식 트리를 이뤄 하나의 trace를 구성한다. M10의 Sentry는 **한 프로세스 안**의 trace만 자동으로 이어준다.
- **컨텍스트 전파(context propagation):** span들을 한 trace로 묶으려면 각 작업이 **같은 traceId와 부모 spanId**를 알아야 한다. 이 식별자를 담은 것을 **trace 컨텍스트**라 하고, Sentry는 이를 `sentry-trace`·`baggage` 두 헤더(W3C traceparent 계열)로 직렬화한다. 이 헤더를 다음 작업에 전달(inject)하고 받는 쪽이 복원(extract)하면 trace가 이어진다.
- **왜 프로세스 경계에서 trace가 끊기나:** trace 컨텍스트는 현재 프로세스의 메모리(비동기 실행 컨텍스트)에만 산다. HTTP 서버 → Kafka → 워커처럼 **다른 프로세스**로 작업이 넘어가면, 받는 프로세스는 그 메모리를 공유하지 않으므로 컨텍스트를 모른 채 **새로운(관계 없는) trace**를 연다. 그래서 컨텍스트를 **메시지에 실어(wire)** 함께 보내야 한다.
- **이 프로젝트의 전파 경로:** HTTP 요청의 trace를 시작점으로, ① **직접 Kafka 발행**(`kafka-event.publisher`)은 발행 직전 `captureTraceHeaders()`로 현재 trace를 헤더로 직렬화해 Kafka 메시지 헤더에 붙이고, ② **워커**(audit·persistence·notification 컨트롤러)는 `kafkaTraceHeaders(ctx)`로 그 헤더를 읽어 `continueTraceFromHeaders(...)`로 복원한 뒤 핸들러를 원본 trace의 자식 span으로 실행한다.
- **Outbox 지연 발행의 특수성 — 컨텍스트를 DB에 저장:** Outbox(§8 참고)는 HTTP 트랜잭션 **안에서 행만 쓰고**, 실제 Kafka 발행은 나중에 **별도 프로세스인 relay 워커**가 한다. 발행 시점엔 원본 HTTP 요청의 메모리 컨텍스트가 이미 사라진 상태다. 그래서 outbox 행을 쓸 때(`prisma-outbox-store`) `captureTraceHeaders()`로 직렬화한 헤더를 **`traceContext` JSON 컬럼에 저장**해 두고, relay가 발행할 때(`relay-outbox.use-case`) 그 컬럼을 읽어 `continueTraceFromHeaders`로 **되살려(continueTrace)** 발행 span을 원본 trace에 붙인다. 즉 끊긴 시간·프로세스 간극을 **DB 컬럼이 다리 역할**로 잇는다.
- **핵심 API:** `Sentry.getTraceData()`(현재 trace를 헤더로 직렬화 = inject) ↔ `Sentry.continueTrace({ sentryTrace, baggage }, fn)`(헤더에서 trace를 복원해 fn을 자식 span으로 실행 = extract). 헬퍼 `captureTraceHeaders`/`continueTraceFromHeaders`/`kafkaTraceHeaders`가 이 둘을 감싸 Kafka 헤더(Buffer)·outbox 컬럼(JSON) 형태로 오가게 한다.
- **best-effort·비침습 원칙:** 모든 헬퍼는 `try/catch`로 감싸, Sentry 비활성(DSN 없음)·직렬화 실패·헤더 없음 등 어떤 경우에도 **빈 맵을 반환하거나 원래 함수(fn)를 그대로 실행**한다. 트레이싱이 실패해도 발행·소비 같은 **업무 경로는 절대 깨지지 않는다**. 관측성은 부가 기능이지 정합성 경로가 아니라는 판단이다.

### 트레이드오프
- **관측성 ↔ 저장 비용·결합:** trace 컨텍스트를 outbox 행에 함께 저장하면 발행이 지연·재시도돼도 trace가 끊기지 않지만, 컬럼 하나만큼의 저장 비용과 "발행 로직이 트레이싱 헤더를 안다"는 약한 결합이 생긴다. 비침습 설계(실패 시 무시)로 결합의 위험은 낮췄다.
- **연결 확인은 DSN이 있어야:** 코드는 DSN 없이 no-op으로 안전하지만, **연결된 trace를 눈으로 보려면** Sentry DSN을 설정해 실제 이벤트를 전송해야 한다. DSN 없이는 "끊기지 않는다"는 것만 코드·테스트로 보장된다.

### 스스로 점검
- [ ] 같은 프로세스 안에서는 Sentry가 span을 자동으로 잇는데, 왜 HTTP→Kafka→워커에서는 trace가 끊기나?
- [ ] Outbox 경로에서만 trace 컨텍스트를 DB 컬럼(`traceContext`)에 저장해야 했던 이유는? 직접 Kafka 발행과 무엇이 다른가?
- [ ] 트레이싱 헬퍼가 전부 `try/catch`로 폴백하도록 만든 이유는 — 반대로 실패 시 예외를 던지게 했다면 어떤 위험이 생기는가?

### 더 팔 키워드

## 8.10 회복탄력성 패턴 — 카카오 OAuth에 타임아웃·재시도·서킷 브레이커·벌크헤드 (M12)

### 개념
- **네 패턴의 역할 분담:** 외부 의존성(카카오)을 호출하는 코드는 "얼마나 기다릴지", "실패하면 다시 할지", "동시에 몇 개까지 보낼지", "계속 실패하는 상대를 언제부터 안 부를지"를 각각 따로 결정해야 한다. 이 네 가지를 하나씩 맡는 것이 **타임아웃**(정해진 시간 안에 응답이 없으면 포기), **재시도**(일시적인 오류를 다시 시도해 흡수), **벌크헤드**(동시 실행 개수를 제한해 한 의존성의 지연이 다른 요청까지 잠식하지 못하게 격리), **서킷 브레이커**(연속 실패가 임계치를 넘으면 회로를 열어 호출 자체를 막고 즉시 실패시키는 fail-fast, 일정 시간 뒤 half-open 상태로 소량만 흘려보내 복구 여부를 탐침)다. 네 패턴은 서로 다른 축의 문제를 다루므로 하나만으로는 부족하고, 한 세트로 감싸 조합해야 한다. **Node.js의 벌크헤드는 스레드 풀이 아니라 세마포어**(동시 실행 허용 개수를 세는 카운터)로 구현된다 — Node는 단일 스레드 이벤트 루프이므로 별도 스레드를 격리하는 것이 아니라, 동시에 진행 중인 비동기 호출 개수 자체를 제한해 이벤트 루프가 한 의존성 대기로 막히는 것을 막는다.
- **멱등성(idempotency)과 재시도 안전성:** **멱등하다**는 것은 같은 요청을 여러 번 반복해도 결과가 한 번 실행한 것과 같다는 뜻이다. 카카오 OAuth의 인가코드(authorization code)는 **1회용**이라 토큰 교환 POST 요청은 멱등하지 않다. 요청이 카카오에 도달해 토큰을 발급했지만 응답이 유실되는 상황을 가정하면, 클라이언트는 실패로 보고 같은 코드로 재시도하지만 카카오는 이미 소모된 코드라며 거절한다. 즉 **재시도가 오히려 로그인 실패를 만든다**. 그래서 토큰 교환에는 재시도를 걸지 않고(`tokenPolicy`는 서킷 브레이커·벌크헤드·타임아웃만 포함), 여러 번 불러도 안전한 프로필 조회 GET에만 재시도를 건다(`profilePolicy`). 일반화하면, 멱등이 보장되지 않는 POST/PATCH 요청은 서버가 `Idempotency-Key` 같은 중복 방지 장치를 제공하지 않는 한 재시도 대상에서 제외해야 한다.
- **일시적(transient) 오류 판별 — `handleAll`이 위험한 이유:** cockatiel은 어떤 예외를 재시도·집계 대상으로 볼지 `handleWhen` 술어로 지정한다. 이때 모든 예외를 무조건 대상으로 삼는 `handleAll`을 쓰면 두 가지 문제가 생긴다. 첫째, 잘못된 인가코드 같은 **4xx(사용자·계약 오류)**까지 재시도하게 되는데, 이런 오류는 몇 번을 다시 보내도 같은 결과라 자원만 낭비한다. 둘째, 4xx가 대량으로 몰리는 상황(예: 클라이언트 버그로 잘못된 요청 폭주)에서 서킷 브레이커가 이를 "카카오 장애"로 오판해 **회로를 열어버려**, 실제로는 멀쩡한 카카오 API를 정상 사용자에게까지 차단하는 결과를 낳는다. 이 프로젝트는 `KakaoApiError.transient`(상태 코드 429·5xx), 네트워크 오류(`TypeError` — fetch의 연결 거부·DNS 실패 등), 시도당 타임아웃(`TaskCancelledError`)만 재시도·서킷 집계 대상으로 좁혔다. 회로가 이미 열렸을 때 던져지는 `BrokenCircuitError`는 의도적으로 집계 대상에서 뺀다 — 이를 포함하면 열린 회로 자체가 다시 실패로 집계되어 복구를 방해하는 순환이 생긴다.
- **wrap 조합 순서의 의미:** cockatiel의 `wrap(a, b, c)`는 `a`가 가장 바깥, 마지막 인자가 가장 안쪽이 되도록 정책을 감싼다. 이 프로젝트는 `retry → circuitBreaker → bulkhead → timeout` 순서로 감싼다. **retry가 breaker 바깥에 있다는 것**은 재시도로 인한 각 시도가 서킷 브레이커에 개별적으로 집계된다는 뜻이고(한 번의 논리적 요청이 여러 번의 실패로 잡혀 회로가 더 빨리 열릴 수 있다), 동시에 **재시도 도중 회로가 열리면 남은 재시도가 즉시 중단**된다는 뜻이다(이미 안 되는 걸 아는데 계속 두드리지 않는다). **timeout이 가장 안쪽(최내곽)에 있다는 것**은 타임아웃이 전체 재시도 묶음이 아니라 **시도 한 번 한 번**에 적용된다는 뜻이다 — 그래야 느린 시도 하나가 재시도 예산을 다 써버리기 전에 빨리 판단하고 다음 시도로 넘어갈 수 있다. 이 조합에서 놓치기 쉬운 함정은 **`AbortSignal` 전달**이다. cockatiel의 타임아웃 정책은 시간 초과 시 `AbortSignal`을 발생시켜 실행 중인 콜백에 취소를 알리는데, 콜백(fetch 호출)이 이 signal을 실제 요청에 넘기지 않으면 "정책은 포기했다고 선언했지만 실제 소켓 연결은 백그라운드에 계속 매달려 있는" 상태가 된다. 그래서 `callKakao`는 `policy.execute(({ signal }) => fetch(url, { signal }))` 형태로 정책이 제공한 signal을 fetch에 그대로 전달한다.
- **fail-fast vs fallback:** 서킷이 열리거나 벌크헤드가 포화되거나 타임아웃이 발생했을 때 취할 수 있는 전략은 크게 **fallback**(대체 데이터나 캐시된 값으로 응답)과 **fail-fast**(대체할 것이 없으니 즉시 실패로 응답)로 나뉜다. 카카오 로그인은 실시간으로 발급받아야 하는 인가코드·토큰·프로필을 다루므로 캐시하거나 대체할 데이터가 존재하지 않는다. 이런 경우 억지로 응답을 지연시키며 재시도를 반복하는 것보다, 실패를 판단할 수 있는 시점에 **503(`AUTH_KAKAO_UNAVAILABLE`)로 즉시 응답**해 클라이언트가 빠르게 재시도 여부를 판단하게 하는 쪽이 옳다.
- **M9 Outbox 재시도와의 구분:** 같은 "재시도"라는 단어를 쓰지만 M9(§8)의 Outbox 재시도와 M12의 카카오 재시도는 설계 축이 다르다. Outbox relay는 **비동기 발행 경로**로, 사용자가 응답을 기다리지 않고 시간 제약이 느슨해 무한 재시도(백오프 + 최종적으로 DLQ 격리)가 합리적인 선택이다. 반면 카카오 OAuth 호출은 **동기 요청 경로**로, 사용자가 로그인 화면에서 응답을 기다리고 있으므로 재시도 횟수와 타임아웃에 엄격한 상한을 두고, 그 상한을 넘으면 지체 없이 실패를 반환해야 한다. 즉 "기다릴 수 있는 배경 작업"과 "사용자가 지켜보는 요청"은 같은 재시도라는 도구라도 다른 예산으로 다뤄야 한다.

### 트레이드오프
- **재시도 범위 축소 ↔ 복구력:** 재시도 대상을 429·5xx·네트워크·타임아웃으로 좁히면 4xx 오탐이나 회로 오작동은 막지만, 그만큼 "혹시 일시적일 수도 있는" 애매한 오류는 재시도되지 않고 바로 실패로 처리된다. 안전한 실패 판별을 우선한 선택이다.
- **fail-fast ↔ 사용자 경험:** 즉시 503을 반환하면 서버 자원은 아끼지만, 사용자 입장에서는 "몇 초 더 기다리면 될 수도 있는" 요청도 실패로 끝난다. 로그인처럼 재시도 비용(다시 로그인 버튼을 누르는 정도)이 낮은 흐름에서는 이 트레이드오프가 fail-fast 쪽으로 기운다.
- **파라미터 튜닝 시점:** 타임아웃·재시도 횟수·브레이커 임계치·벌크헤드 크기 6종은 모두 env로 조정 가능하지만, 현재 값은 실제 카카오 트래픽을 측정하기 전의 잠정값이다. 과도하게 낮으면 정상 상황도 실패로 몰고, 과도하게 높으면 장애 감지가 늦어진다 — M7·M11처럼 실측 후 조정이 필요한 항목이다.
- **브레이커 임계 × 재시도 배수 효과(놓치기 쉬운 상호작용):** 재시도(`retry`)가 공유 `breaker` 바깥에 있으므로, 프로필 GET 한 번의 논리적 실패가 브레이커의 연속 실패 카운트를 **최대 (재시도 횟수 + 1)회** 소모한다(기본 `maxAttempts=3`이면 초기 시도 포함 최대 4회). 그래서 `breakerThreshold=5`라도 실제로는 프로필 로그인 **1~2회 실패만으로** 회로가 열려, 브레이커를 공유하는 토큰 교환까지 즉시 503으로 차단될 수 있다 — 파라미터 이름이 주는 인상("연속 5회 실패")과 체감 동작이 다르다. 이는 실측과 무관하게 존재하는 설계 특성이므로, 튜닝 시 `breakerThreshold`는 재시도 배수를 감안해(대략 `임계치 ≳ (재시도 횟수 + 1) × 허용할 논리적 실패 수`) 잡아야 한다. 재시도를 breaker 안쪽에 두면 이 배수 효과는 사라지지만, 대신 "재시도 도중 회로가 열리면 남은 재시도 즉시 중단" 이점을 잃는다.
- **느린(죽지 않은) 의존성의 꼬리 지연 — 전체 시간 예산의 부재:** 경량 하네스로 실측(가짜 카카오에 지연·에러 주입, [`load/results/m12-resilience.md`](../../load/results/m12-resilience.md))한 결과, 정상·장애·간헐에러 구간에선 기본값이 안전했다(정상 오탐 0, 장애 fail-fast 503 평균 8ms, 간헐 30%에서 브레이커 미개방). 반면 **카카오가 느리지만 죽지는 않은** 구간에서 프로필 재시도(최대 4회)가 per-attempt 타임아웃(3s)과 곱해져 **p95가 ~15초까지 스택**됐다. 이는 합성 지연 분포와 무관한 **구조적** 문제로, 재시도 정책에 **전체 시간 예산(total timeout budget)**이 없기 때문이다(위키 팀룰 "전체 재시도 과정에 타임아웃 예산"). 시도당 타임아웃만으로는 "느림"에서 꼬리를 못 막는다 — 죽은 의존성(장애)엔 브레이커가 fail-fast하지만, 느린 의존성엔 브레이커가 잘 안 열려(성공이 섞여 연속 실패가 안 쌓임) 재시도가 조용히 스택된다. **해결(구현됨):** `profilePolicy`를 바깥 timeout으로 한 번 더 감싸(`wrap(totalTimeout, retry, breaker, bulkhead, perAttemptTimeout)`) 재시도 스택 전체에 벽시계 상한을 걸었다(`KAKAO_TOTAL_TIMEOUT_MS`, 기본 8000ms). 격리 실측 결과 예산 없이 ~24s이던 프로필 꼬리가 예산 8s로 끊겼다(단위 테스트로도 250ms 예산 → 252ms 결정적 검증). 토큰 교환은 재시도가 없어 시도당 타임아웃으로 이미 상한이 걸리므로 대상 아님. **핵심 통찰:** 브레이커는 "죽은" 의존성에, 총 예산은 "느린" 의존성에 각각 상한을 건다 — 둘은 다른 실패 양상을 덮는 보완재다.

### 스스로 점검
