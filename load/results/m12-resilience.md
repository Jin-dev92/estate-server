# M12 — 회복탄력성 파라미터 실측 (카카오 OAuth)

## 방법과 한계

- **왜 k6 엔드투엔드가 아니라 하네스인가:** 회복탄력성 계층은 "카카오가 오작동할 때만" 작동한다. 실제 카카오로는 1회용 인가코드·외부 rate limit 때문에 부하를 줄 수 없고 장애를 주입할 수도 없다. 그래서 실제 프로덕션 경로(`KakaoOAuthClient.exchangeAndFetch` = cockatiel 정책 조립 + 503 매핑)를 그대로 구동하되, 단위 테스트와 동일하게 `global.fetch`에 **가짜 카카오(지연·에러 주입)**를 물려 카카오 장애 분포를 흉내낸다. 앱·DB·Redis·Kafka 없이 튜닝 대상(정책 계층)만 정밀 측정한다. 하네스: [`load/resilience/harness.ts`](../resilience/harness.ts), 실행 `npx ts-node load/resilience/harness.ts`.
- **think time이 필수인 이유(측정 함정):** 회로가 열리면 이후 로그인은 즉시 503(≈0ms)이다. think time 없이 닫힌 루프로 돌리면 이 즉시-503을 초당 수만 건 회전시켜 `p95=0`·`throughput 10만/s` 같은 무의미한 수를 만든다(초기 실험에서 실제로 관측). 실제 사용자는 로그인 1번 후 텀이 있으므로, 로그인 사이에 think time(120ms)을 둬 페이싱한다(M8 부하테스트 think time과 같은 원리).
- **한계(중요):** 지연·에러율은 **합성값**이다. 실제 카카오 트래픽의 지연 분포·장애 패턴이 아니다. 따라서 아래 숫자는 **"파라미터가 각 상황에서 어떻게 동작하는가(행동·트레이드오프)"**를 드러낼 뿐, "이 값이 정답"을 증명하지 않는다. 정확한 값 튜닝은 프로덕션 카카오 지연 p50/p95 데이터가 있어야 가능하다(§결론). — M8에서 "숫자는 머신 한계이지 절대 한계가 아님"이라 적은 것과 같은 성격의 통제 실험이다.
- 조건: `concurrency=20`, `duration=2500ms/cell`, `think=120ms`. 지표 — p95(all)=전체 로그인, p95(ok)=성공 로그인만(들어간 사용자 체감), 503평균=503 응답 평균 지연(낮을수록 fail-fast). 현행 기본값 = timeout 3000 · retry 3 · breaker 임계 5 · half-open 10000 · bulkhead 10/20.

## 관찰

- **정상:** 오탐(spurious open) 0, 100% 성공. p95 407ms는 2회 순차 호출(각 60~140ms) + 벌크헤드(동시 10)로 인한 큐잉이 더해진 값으로, 정상 상황에서 회복탄력성 계층의 오버헤드가 과하지 않음을 보여준다.
- **장애(fail-fast 검증):** 100% 5xx에서 회로가 곧 열리고(open 1회), 이후 로그인은 **503 평균 8ms**로 즉시 실패한다. 이것이 fail-fast의 목표다 — 죽은 카카오를 매번 풀 타임아웃까지 기다리지 않는다. threshold 5와 2는 결과가 사실상 동일(둘 다 빠르게 open)해, **임계를 낮춰도 장애 대응 이득은 없다**.
- **간헐에러(브레이커 배수 가드 검증):** 30% 5xx에서 default(threshold 5)는 **회로를 열지 않는다**(open 0). 재시도가 브레이커 바깥이라 실패가 배수로 집계됨에도, 임계 5가 30% 산발 실패를 견뎠다 — `threshold ≥ retry+1` 가드(설정 검증에 반영)가 실제로 오탐을 막는 것을 확인. threshold를 8로 올려도 이득 없음(이미 안 열림). retry는 성공률을 사준다: retry 3(72%) vs retry 1(63%), 대신 retry 3은 p95(ok) 839ms로 retry 1(525ms)보다 지연이 높다.
- **느림(가장 중요한 발견):** default(timeout 3000, retry 3)는 80% 성공이지만 **p95가 14.9초**다. 프로필 GET 재시도(최대 4회)가 각각 최대 3초 타임아웃 + 백오프로 스택되기 때문이다 — **전체 시간 예산(total timeout budget)이 없어** 느린(죽지 않은) 카카오에서 꼬리 지연이 폭증한다. 타임아웃을 1500으로 낮추면 타임아웃이 브레이커에 다수 집계돼 회로가 열리고, 성공률은 2.4%로 급락하지만 503 평균 491ms로 빠르게 실패한다.
- **느림+고동시성(벌크헤드 트레이드오프):** 동시 40(벌크헤드 용량 30 초과)에서 default는 초과분을 즉시 거절(87.5% 503, 503평균 0ms = 즉시)해 앱을 보호한다. 벌크헤드를 30/40(용량 70)으로 키우면 전원 수용(100% 성공)하나 p95 3.6초로 느려진다 — "거절로 보호" vs "수용하고 지연"의 전형적 트레이드오프.

## 결론과 튜닝 판단

1. **정상·장애·간헐에러 구간에서 현행 기본값은 안전하다.** 정상에서 오탐 0, 장애에서 fail-fast(8ms), 간헐 30%에서 브레이커가 성급히 열리지 않음(배수 가드가 버팀). 이 세 구간을 근거로 **기본값을 바꿀 이유는 없다.**
2. **가장 큰 리스크는 "느린(죽지 않은) 카카오"에서의 꼬리 지연이다.** retry가 per-attempt 타임아웃과 곱해져 p95가 스택된다. 이는 합성 지연 분포와 무관하게 **구조적**이다(전체 시간 예산 부재). 로그인은 사용자가 화면에서 기다리는 동기 경로라 십수 초는 사실상 실패다. → 이 리스크를 **전체 시간 예산(total timeout budget)으로 해소했다**(아래 [§총 예산 도입·검증](#총-예산total-timeout-budget-도입검증-2026-07-07)). `profilePolicy`를 바깥 timeout으로 한 번 더 감싸(`wrap(totalTimeout, retry, breaker, bulkhead, perAttemptTimeout)`) 재시도 스택 전체에 벽시계 상한을 건다(위키 팀룰 "전체 재시도 과정에 타임아웃 예산").
3. **합성 데이터로 정확한 새 값을 확정하지 않는다.** timeout·retry·bulkhead의 "정답"은 프로덕션 카카오의 실제 지연 p50/p95와 예상 동시 로그인 수에 달렸다. 그 데이터가 확보되면 이 하네스에 실제 분포를 넣어 재측정한다(하네스는 재사용 가능한 도구로 커밋).
4. **결정:** 현행 env 기본값 6종을 **잠정 기준값으로 유지**하되, "느림 꼬리 지연" 리스크와 "전체 시간 예산" 개선안을 문서화한다. 벌크헤드·임계 확대는 예상 피크 동시성·실측 없이 바꾸면 오히려 보호를 약화하므로 보류.

## 총 예산(total timeout budget) 도입·검증 (2026-07-07)

위 결론 2의 "느림 꼬리 지연"을 해소하기 위해 `profilePolicy`에 **바깥 총 타임아웃**을 추가했다(`KAKAO_TOTAL_TIMEOUT_MS`, 기본 8000ms). 효과를 격리 측정하려고 **토큰은 빠르게 성공·프로필은 매 시도가 타임아웃 초과**하도록 하고, 브레이커는 비활성(임계 100 — 회로가 먼저 열리면 예산 효과가 가려진다)으로 뒀다.

| 변형 | 성공 | 503 | p95(all) | 503평균 | 서킷open |
|---|---|---|---|---|---|
| budget 8000 (default) | 0% | 100% | 8193ms | **8100ms** | 0 |
| budget off (60000) | 0% | 100% | 24178ms | **22652ms** | 0 |

- **예산 8000이 재시도 스택을 ~8초에서 끊는다.** 예산이 없으면(60000) 프로필 4시도(초기+재시도 3) × 3초 타임아웃 + 지수 백오프가 쌓여 **~24초**까지 지연된다. 예산이 이를 약 1/3로 줄인다. 결정적 검증은 단위 테스트(총 예산 250ms → 실측 252ms에서 끊김, `kakao-resilience.spec.ts`).
- 이때 성공률은 둘 다 0%다(프로필이 애초에 항상 타임아웃이라 성공 불가) — 예산은 "성공을 늘리는" 게 아니라 **실패를 빨리 판정**하는 장치다. 실전에선 브레이커가 먼저 열려 보호하지만, 브레이커가 안 열리는 경계(간헐적 느림)에서 예산이 마지막 상한이 된다.
- 토큰 교환은 재시도가 없어 시도당 타임아웃(3s)으로 이미 상한이 걸리므로 예산 대상이 아니다.

> 참고(변동성): 간헐에러(30%) 구간에서 default(threshold 5·retry 3)는 회로가 **열릴 때도, 안 열릴 때도** 있다(재측정 시 관측). 배수(retry+1=4) 때문에 임계 5가 경계값이라 5-연속-실패가 확률적으로 발생한다. `threshold ≥ retry+1` 가드는 최소선일 뿐, 실데이터로 여유를 더 두는 게 안전하다.

## 후속 과제

- 프로덕션 카카오 지연·장애 실데이터 확보 후 이 하네스로 재측정 → 6종+1(총 예산) 값 확정. 특히 총 예산 8000ms·간헐에러 임계는 실측으로 좁힐 여지가 있다.
- (선택) k6 엔드투엔드: 카카오 URL을 env로 빼 장애주입 스텁을 물리면, HTTP·앱 오버헤드까지 포함한 검증 가능(Docker 필요).
