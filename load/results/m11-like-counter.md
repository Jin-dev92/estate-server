# M11 — 좋아요 카운터 전환 전/후 측정

- 조건: 로컬 단일 머신(앱+PG+Redis+Kafka 동시 구동), 글 50개, `read-posts.js` PROFILE=load(20VU, think 1s), 웜업 1회 후 측정. 절대치가 아니라 **동일 조건 상대 비교**용.
- before = 라이브 `COUNT(*)`(GROUP BY IN(50)), after = Redis 카운터(미스 시 COUNT 재구축).

| 글당 좋아요 | 구현 | p95 | RPS | 에러율 |
|---|---|---|---|---|
| 0 | COUNT (before) | 10.61ms | 15.87/s | 0.00% |
| 200 | COUNT (before) | 19.53ms | 15.84/s | 0.00% |
| 2000 | COUNT (before) | 73.46ms | 15.19/s | 0.00% |
| 0 | Redis 카운터 (after) | — | — | — |
| 200 | Redis 카운터 (after) | — | — | — |
| 2000 | Redis 카운터 (after) | — | — | — |

## 관찰 (before)

- 볼륨이 늘어날수록 p95가 뚜렷하게 증가한다: 0개(10.61ms) → 200개(19.53ms, 약 1.8배) → 2000개(73.46ms, 0 대비 약 6.9배). 글당 좋아요 수가 10배 늘 때마다(200→2000) p95도 약 3.8배 늘어, 좋아요 총량에 비례해 `GROUP BY` 집계 비용이 커지는 패턴이 보인다.
- RPS(iterations rate)는 세 볼륨에서 15.87 → 15.84 → 15.19로 거의 변화가 없다. think time(1s)이 지배적이라 처리량 저하보다는 응답 지연(p95) 증가로 비용이 드러난다 — 20VU·1s think 조건에서는 동시성 여유가 있어 RPS caps가 아닌 latency 관점에서 병목이 관찰됨.
- 웜업(smoke, 1VU) 단계에서도 동일한 경향이 선행 관찰됨: 볼륨 0/200/2000에서 p95가 각각 21.98ms/17.78ms/62.66ms로, 2000 볼륨에서 이미 GROUP BY 비용이 드러난다(0/200 웜업은 VU=1·표본 30개라 노이즈가 커서 본 측정만 표에 기록).
- 에러율은 세 볼륨 모두 0.00%(checks 100% 성공), k6 임계값(`p(95)<300ms`, `rate<0.01`)도 모두 통과 — 로컬 단일 머신·50개 글 조건에서는 2000개 좋아요까지도 임계값 내에 있으나, 응답시간이 볼륨에 선형 이상으로 증가하는 추세는 이후 볼륨을 더 키우면(예: 글당 좋아요 수만·수백만) 임계값을 넘길 수 있음을 시사한다.
- 앱 로그(`/tmp/m11-app.log`)에는 측정 구간 동안 에러 없음(부팅 시 Kafka `GroupCoordinator` 일시 에러 1건은 정상적인 컨슈머 조인 재시도로, 이후 정상 join 확인됨).
