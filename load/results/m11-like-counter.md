# M11 — 좋아요 카운터 전환 전/후 측정

- 조건: 로컬 단일 머신(앱+PG+Redis+Kafka 동시 구동), 글 50개, `read-posts.js` PROFILE=load(20VU, think 1s), 웜업 1회 후 측정. 절대치가 아니라 **동일 조건 상대 비교**용.
- before = 라이브 `COUNT(*)`(GROUP BY IN(50)), after = Redis 카운터(미스 시 COUNT 재구축).

| 글당 좋아요 | 구현 | p95 | RPS | 에러율 |
|---|---|---|---|---|
| 0 | COUNT (before) | 10.61ms | 15.87/s | 0.00% |
| 200 | COUNT (before) | 19.53ms | 15.84/s | 0.00% |
| 2000 | COUNT (before) | 73.46ms | 15.19/s | 0.00% |
| 0 | Redis 카운터 (after) | 12.20ms | 15.83/s | 0.00% |
| 200 | Redis 카운터 (after) | 12.39ms | 15.95/s | 0.00% |
| 2000 | Redis 카운터 (after) | 13.27ms | 15.95/s | 0.00% |

## 관찰 (before)

- 볼륨이 늘어날수록 p95가 뚜렷하게 증가한다: 0개(10.61ms) → 200개(19.53ms, 약 1.8배) → 2000개(73.46ms, 0 대비 약 6.9배). 글당 좋아요 수가 10배 늘 때마다(200→2000) p95도 약 3.8배 늘어, 좋아요 총량에 비례해 `GROUP BY` 집계 비용이 커지는 패턴이 보인다.
- RPS(iterations rate)는 세 볼륨에서 15.87 → 15.84 → 15.19로 거의 변화가 없다. think time(1s)이 지배적이라 처리량 저하보다는 응답 지연(p95) 증가로 비용이 드러난다 — 20VU·1s think 조건에서는 동시성 여유가 있어 RPS caps가 아닌 latency 관점에서 병목이 관찰됨.
- 웜업(smoke, 1VU) 단계에서도 동일한 경향이 선행 관찰됨: 볼륨 0/200/2000에서 p95가 각각 21.98ms/17.78ms/62.66ms로, 2000 볼륨에서 이미 GROUP BY 비용이 드러난다(0/200 웜업은 VU=1·표본 30개라 노이즈가 커서 본 측정만 표에 기록).
- 에러율은 세 볼륨 모두 0.00%(checks 100% 성공), k6 임계값(`p(95)<300ms`, `rate<0.01`)도 모두 통과 — 로컬 단일 머신·50개 글 조건에서는 2000개 좋아요까지도 임계값 내에 있으나, 응답시간이 볼륨에 선형 이상으로 증가하는 추세는 이후 볼륨을 더 키우면(예: 글당 좋아요 수만·수백만) 임계값을 넘길 수 있음을 시사한다.
- 앱 로그(`/tmp/m11-app.log`)에는 측정 구간 동안 에러 없음(부팅 시 Kafka `GroupCoordinator` 일시 에러 1건은 정상적인 컨슈머 조인 재시도로, 이후 정상 join 확인됨).

## 관찰 (after)

- 측정 조건: 볼륨 전환마다 `redis-cli FLUSHALL`로 캐시·카운터를 모두 비운 뒤, 웜업(smoke, 1VU) 1회가 `list` 요청의 `readMany` 백필 경로를 통해 50개 게시글의 카운터를 전부 재구축한다. 본 측정(load, 20VU)은 이렇게 **이미 워밍된 카운터**를 읽는 조건이며, cold-miss 비용은 표에 포함되지 않는다(웜업 자체의 p95는 아래에 별도 기록).
- p95는 볼륨 0/200/2000에서 12.20ms → 12.39ms → 13.27ms로 거의 평평하다. 글당 좋아요가 0→2000으로 늘어도 p95 증가폭은 약 1.07ms(약 8.8%)에 불과해, before의 6.9배(10.61→73.46ms) 증가와 뚜렷이 대비된다.
- RPS는 세 볼륨에서 15.83 → 15.95 → 15.95로 안정적이며, before와 마찬가지로 think time이 지배적이라 처리량보다는 지연 시간에서 차이가 드러나는 조건이다.
- 에러율은 세 볼륨 모두 0.00%(checks 100% 성공), k6 임계값(`p(95)<300ms`, `rate<0.01`) 모두 통과. 앱 로그(`/tmp/m11-after.log`)에도 측정 구간 에러 없음.
- 웜업(smoke) 단계 참고 수치: 0/200/2000에서 p95가 각각 25.17ms/13.25ms/29.10ms로, 카운터 백필(SET NX + `readMany`의 COUNT 재구축)이 섞인 1회성 요청이라 노이즈가 크다(VU=1·표본 30개). 본 측정에는 포함하지 않는다.

## 결론

- **before(라이브 COUNT)는 볼륨에 비례해 p95가 상승했다**: 0→2000 좋아요 구간에서 10.61ms → 73.46ms로 약 6.9배 증가. 반면 **after(Redis 카운터, 워밍 상태)는 같은 구간에서 12.20ms → 13.27ms로 사실상 평평**하다(약 8.8% 증가에 그침). 즉 카운터 전환의 효과는 "절대값을 항상 더 빠르게 만든다"가 아니라 **"응답 시간을 좋아요 총량과 무관하게 만든다"**는 데 있다 — `COUNT ... GROUP BY`가 좋아요 행 수에 선형 이상으로 비례해 커지는 반면, Redis `GET`/`MGET` 성격의 조회는 행 수와 무관한 O(1) 조회이기 때문이다.
- 저볼륨(0/200) 구간만 보면 after(12.20~12.39ms)가 before(10.61~19.53ms)보다 항상 우세하다고 단정하긴 어렵다 — 볼륨 0에서는 before(10.61ms)가 오히려 after(12.20ms)보다 낮게 나왔는데, 이는 Redis 조회(네트워크 왕복 1회 추가) 자체의 고정 비용이 붙기 때문으로 해석된다. 즉 **좋아요 수가 아주 적을 때는 COUNT가 카운터보다 오히려 미세하게 유리할 수 있다** — 이번 실측 규모(로컬 단일 머신, 글 50개)에서는 그 차이가 왕복 지연 1~2ms 수준으로 작아 실질적 영향은 미미하다.
- 교차점은 200과 2000 사이에 있다: 200개에서는 before(19.53ms)와 after(12.39ms)가 이미 역전되어 after가 우세하고, 2000개에서는 그 격차가 73.46ms vs 13.27ms로 약 5.5배까지 벌어진다. **글당 좋아요 수가 많아질수록(볼륨이 커질수록) 카운터 전환의 이득이 커지는 구조**이며, 이 실험 규모(최대 2000개/글)에서 이미 그 경향이 뚜렷이 관찰된다.
- drift(카운터와 실제 행 수의 불일치) 수용 근거: 카운터 증감은 좋아요/취소 트랜잭션이 **커밋된 이후에만** best-effort로 수행되고, 실패해도 요청 자체는 성공 처리된다(카운터가 진실 원천(source of truth)이 아니라 `PostLike` 테이블의 **파생 캐시**이기 때문에 가능한 설계). 따라서 인프라 장애 등으로 카운터가 일시적으로 실제 값과 어긋날 수 있음을 설계상 감수한다.
  - 치유 경로 1(TTL): 카운터 키에 3600초 TTL을 부여해, 드리프트가 누적되더라도 최대 1시간 내에 키가 만료되고 다음 조회 시 `COUNT`로 재구축되며 자연 정정된다.
  - 치유 경로 2(SET NX 백필): 카운터 키가 없을 때(TTL 만료 또는 최초 조회)는 `readMany`/`readOne`이 COUNT로 실제 값을 구해 `SET NX`로 백필한다. `NX`(키가 없을 때만 설정)를 쓰는 이유는 동시 요청 다수가 동시에 미스를 겪어도(레이스 컨디션) 단 하나의 SET만 반영되게 해 백필 경합 시 값이 서로 다른 요청으로 덮어써지는 것을 막기 위함이다.
  - 존재-시-증감(exists-then-incr) Lua 스크립트: 키가 존재할 때만 원자적으로 INCR/DECR한다. 키가 아직 백필되지 않은 상태(0으로 취급되어야 할 "없음")에서 증감이 먼저 들어오면 실제 값과 다른 값으로 초기화되는 "0→1 함정"을 막기 위한 장치다(§2.5, Task 3 구현).
